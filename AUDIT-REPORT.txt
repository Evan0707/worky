  3.2. PROJET "DELETE" = Soft Delete
  ----------------------------------
  IMPORTANT : project.delete ne fait PAS de suppression reelle.
  Il fait : `data: { status: "DONE" }` (project.ts:160)

  Ca veut dire :
  - Les donnees du projet (photos, temps, materiaux, factures) restent intactes
  - Le projet passe juste en statut "Termine"
  - Il n'y a AUCUN moyen via l'API de supprimer reellement un projet
  - Seule la suppression du User (cascade) supprimerait les projets

  C'est un choix delibere (pas de perte de donnees accidentelle),
  mais l'UI affiche "Supprimer le chantier" avec un message "irreversible"
  ce qui est trompeur — c'est en fait un archivage.


  3.3. Quand un User Rejoint une Team
  ------------------------------------
  Quand un user accepte une invitation (team.acceptInvite) :

  1. La procedure verifie qu'il n'est pas deja dans une team
  2. Elle verifie qu'il ne possede pas deja une team
  3. Elle cree un TeamMember et marque l'invitation comme acceptee

  CONSEQUENCE : Le user qui rejoint une team garde ses propres donnees
  (projets, photos, etc. sous son propre artisanId). MAIS :

  - getArtisanContext() va maintenant retourner l'artisanId du OWNER de la team
  - Donc le user ne verra PLUS ses anciens projets (filtre artisanId != lui)
  - Ses anciens projets deviennent des donnees orphelines invisibles
  - Si il quitte la team, il retrouvera ses anciens projets

  [BUG] Il n'y a aucun avertissement dans l'UI ni dans l'API
  que rejoindre une team "cache" les projets personnels existants.

  RECOMMANDATION :
  - Soit transferer les projets au nouveau artisanId lors du join
  - Soit empecher de rejoindre si le user a des projets existants
  - Soit afficher un warning explicite


  3.4. Quand un Member Quitte la Team
  ------------------------------------
  team.leave (ligne 404) : Supprime le TeamMember.
  - Les donnees creees par ce member (photos, temps, materiaux) RESTENT
    car elles sont liees au Project (pas au User directement)
  - Le champ createdById des records pointe toujours vers le user
  - Les taches assignees au member restent assignees (assigneeId)

  [OK] Pas de perte de donnees, mais les references createdBy/assignee
  pointent vers un user qui n'a plus acces — cosmétique seulement.

  3.5. team.removeMember
  ----------------------
  Meme comportement que leave — supprime le TeamMember.
  Les ProjectAssignment du member NE SONT PAS supprimees automatiquement
  (pas de cascade TeamMember → ProjectAssignment car pas de relation directe).

  [BUG MINEUR] Apres retrait d'un membre, ses ProjectAssignment persistent
  en base. Ce n'est pas un probleme de securite (getArtisanContext empechera
  l'acces) mais c'est de la donnee orpheline.

  8. PROBLEMES SCHEMA / INTEGRITE DES DONNEES
================================================================================

  8.1. BLOQUANT : ProjectNote.createdById et Task.createdById non-nullable
       sans onDelete (CRITIQUE)
  -----------------------------------------------------------------------
  - ProjectNote.createdById (schema.prisma:279) est `String` (non nullable)
    et n'a pas de `onDelete` → Prisma default = Restrict
  - Task.createdById (schema.prisma:329) est `String` (non nullable)
    et n'a pas de `onDelete` → Prisma default = Restrict

  CONSEQUENCE : Si un User est supprime (au niveau DB), la suppression
  va ECHOUER avec une erreur FK constraint si ce user a cree des notes
  ou des taches. Toutes les autres relations createdBy (Photo, TimeEntry,
  Material, Invoice, Message) sont nullable (String?) donc SetNull par defaut.

  RECOMMANDATION : Changer en `String?` pour coherence, ou ajouter
  `onDelete: SetNull` dans la relation.

  8.2. Pas d'endpoint de suppression de compte User
  --------------------------------------------------
  La page legale (legal.json) mentionne le droit de suppression de compte,
  mais aucun endpoint n'existe dans userRouter. Quand ce sera implemente,
  le probleme 8.1 bloquera la suppression.

  8.3. Pas de cleanup UploadThing sur cascade
  --------------------------------------------
  Quand les photos sont supprimees en cascade (User → Project → Photo),
  les fichiers dans UploadThing ne sont PAS nettoyes. Photo.key existe
  pour ca mais aucun hook de suppression ne l'utilise.

  RECOMMANDATION : Ajouter un webhook ou middleware Prisma pour appeler
  UploadThing deleteFiles() quand des photos sont supprimees.
