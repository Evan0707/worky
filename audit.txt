  3. SÉCURITÉ
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ──── CRITIQUE ────

  [SEC-01] IDOR dans invoice.create
           Utilise ctx.session.user.id au lieu de getArtisanContext()
           → Un membre d'équipe ne peut pas créer de factures du tout
           → Incohérence avec toutes les autres mutations invoice qui
             utilisent getArtisanContext()
           Fichier : src/server/api/routers/invoice.ts ~ligne 70

  [SEC-02] Pas de vérification PRO sur invoice.create et createFromProject
           La landing page et le billing décrivent la facturation comme PRO,
           requirePro() existe dans trpc.ts mais n'est jamais appelé dans
           le router invoice. Un utilisateur FREE peut créer des factures
           illimitées via l'API tRPC directement.

  [SEC-03] /api/webhooks/pdp non authentifié sans TIIME_WEBHOOK_SECRET
           Si la variable d'env n'est pas définie, n'importe qui peut POST
           sur cet endpoint et changer le statut de n'importe quelle facture.
           Fichier : src/app/api/webhooks/pdp/route.ts

  [SEC-04] Race condition sur le numéro de facture
           Deux requêtes concurrentes → même count+1 → même numéro →
           crash @unique Prisma → erreur 500 opaque pour l'utilisateur.

  ──── MOYEN ────

  [SEC-05] Création d'équipe non verrouillée au plan PRO
           team.create n'appelle pas requirePro() → un utilisateur FREE
           peut créer une équipe via l'API.

  [SEC-06] JWT plan staleness après upgrade Stripe
           Le plan est écrit dans le JWT uniquement au sign-in.
           Après un upgrade PRO via webhook Stripe, le JWT reste stale
           jusqu'à déconnexion/reconnexion. L'utilisateur paie mais ne
           voit pas les features PRO immédiatement.

  [SEC-07] markClientViewed (publicProcedure) sans rate limiting
           N'importe qui avec un shareToken peut spammer ce endpoint.

  [SEC-08] Signature base64 stockée en JSON sans limite de taille
           Le PNG complet de la signature est dans ClientAction.payload.
           Peut bloater la DB et les réponses API.

  ──── FAIBLE ────

  [SEC-09] photo.updateOrder ne vérifie pas que tous les IDs photos
           appartiennent au projectId donné (mais l'artisan est vérifié).
  [SEC-10] invoice-template.ts injecte des champs utilisateur dans du HTML
           via template literals sans échappement (risque limité car
           traitement server-side uniquement par WeasyPrint).
  [SEC-11] NEXT_PUBLIC_APP_URL default à http://localhost:3000
           → tous les liens partagés pointent vers localhost si non défini.
