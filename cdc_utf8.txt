01  Contexte &amp; Vision produit
1.1 Problème identifié
Les artisans du bâtiment gèrent leurs chantiers via des outils inadaptés : carnets papier, photos WhatsApp, SMS, tableurs Excel. Ce mode de fonctionnement génère :

Perte de temps administrative estimée à 6-8h par semaine
Litiges fréquents avec les clients (pas de trace écrite des étapes)
Aucun suivi de rentabilité par chantier (heures réelles vs estimées)
Non-conformité imminente à la facturation électronique obligatoire (sept. 2027 en France, 2026-2027 en Belgique/Allemagne/Espagne)
Problème identique dans toute l'Europe - les artisans allemands, belges, espagnols font face aux mêmes enjeux

1.2 Solution - OpenChantier
OpenChantier est un SaaS web-first (mobile PWA) de gestion de chantier conçu pour l'artisan solo et la petite équipe. Il est pensé dès le départ pour être multilingue et multi-pays.

Gestion des chantiers avec photos géolocalisées et horodatées
Partage temps réel client via magic link sécurisé (sans compte client)
Saisie heures &amp; matériaux  devis &amp; factures auto
Conformité facturation électronique par pays (Factur-X FR, ZUGFeRD DE, Peppol BIS EU)
Interface disponible en fr-FR, en-GB, de-DE, es-ES dès v1.0

02  Internationalisation (i18n)  - Section nouvelle v2
2.1 Stratégie i18n globale
OpenChantier utilise next-intl comme solution i18n. Toutes les chaînes de caractères de l'interface sont externalisées dès le premier commit. Il n'y a aucune chaîne en dur dans le code - zéro exception.

Contrainte LLM absolue : Ne JAMAIS écrire de texte d'interface directement dans le JSX/TSX. Toujours utiliser useTranslations() (Client Component) ou getTranslations() (Server Component) depuis next-intl. Exemple : const t = useTranslations('projects'); return &lt;h1&gt;{t('title')}&lt;/h1&gt;. Cela s'applique aussi aux messages d'erreur tRPC côté client.

2.4 Détection et routage de la locale
URL : /{locale}/... - ex: /fr-FR/chantiers, /de-DE/projekte
Détection automatique via Accept-Language header si l'utilisateur arrive sur /
Locale de l'artisan persistée en DB (User.locale) - utilisée pour les emails Resend
La vue client magic link /c/[token] détecte la locale via Accept-Language (pas de préfixe URL pour le client)
next-intl middleware configuré dans src/middleware.ts AVANT le middleware auth NextAuth

2.5 Internationalisation des données métier
Formats numériques et monétaires
Utiliser Intl.NumberFormat avec la locale de l'utilisateur pour tous les montants
Symbole monétaire configurable dans User.currency (EUR par défaut - GBP pour en-GB)
Séparateur décimal : virgule en fr-FR/de-DE, point en en-GB
Tous les prix en DB stockés en centimes (integer) - jamais en float

Formats de dates
Toutes les dates stockées en UTC en DB (timestamptz Prisma)
Affichage via Intl.DateTimeFormat avec la locale utilisateur
Format court : 14/03/2026 (fr-FR), 03/14/2026 (en-GB), 14.03.2026 (de-DE)
Ne jamais utiliser date-fns directement dans les composants UI - passer par un helper formatDate(date, locale)

Contenu des emails
Les templates React Email ont une locale prop
Resend envoie dans la langue de l'artisan (User.locale)
Les emails de relance facture sont dans la langue du client si clientLocale est défini, sinon locale artisan

Facturation électronique multi-pays
Le format de facture généré dépend du User.country et User.vatScheme
FR + assujetti TVA  Factur-X EN 16931 via microservice Python
DE + assujetti TVA  ZUGFeRD 2.x (compatible Factur-X) via même microservice
BE / NL / ES  Peppol BIS 3.0 via PDP partenaire compatible Peppol
Franchise TVA (micro-entrepreneur FR)  PDF classique conforme + champ mention légale auto

2.6 SEO multilingue
generateMetadata() Next.js génère hreflang pour chaque locale
Sitemap.xml inclut toutes les URL localisées
Open Graph et Twitter Card localisés
Blog d'articles par locale : /fr-FR/blog, /de-DE/blog - contenu distinct, pas de traduction automatique

03  Utilisateurs &amp; Rôles
3.1 Rôle Artisan
L'artisan est le titulaire du compte. Nouvelles données en v2 : locale, country, currency, vatScheme.

Création et gestion de ses chantiers (limite plan FREE)
Upload photos géolocalisées depuis navigateur ou mobile
Saisie heures &amp; matériaux, génération devis &amp; factures
Partage client via magic link
Choix de la langue de l'interface (indépendant de l'URL - persisté en DB)
Configuration pays + régime TVA (impacte le format de facturation électronique)

3.2 Rôle Client (sans compte)
Accède à la vue chantier via magic link - interface dans sa langue (Accept-Language)
Lecture seule : photos, avancement, notes
(PRO) Validation étape, commentaire, signature devis

Contrainte LLM : La réponse tRPC getByToken() ne doit jamais exposer artisanId, clientEmail, stripeCustomerId, User.locale, User.country ou tout autre champ du profil artisan. Utiliser un select Prisma explicite limité aux champs de la vue publique.

3.3 Rôle Admin
Backoffice protégé par ADMIN_SECRET (Basic Auth middleware Edge)
Métriques usage, gestion manuelle abonnements Stripe, logs Sentry
Vue par pays/locale pour suivre la distribution géographique

5.5 Sécurité
Middleware Next.js : toutes les routes /[locale]/(dashboard)/* protégées via auth()
tRPC protectedProcedure : vérifie ctx.session - lève UNAUTHORIZED si absent
Isolation données : chaque query Prisma filtre par artisanId = ctx.session.user.id
Magic link : UUID v4, rate-limit 20 req/min par IP via Edge middleware
Webhooks Stripe : vérification signature stripe.webhooks.constructEvent()
Upload : whitelist MIME types (image/jpeg, image/png, image/webp)
CSP headers : next.config.js - bloquer inline scripts sauf next-intl
HSTS : header Strict-Transport-Security via Vercel

07  Exigences non-fonctionnelles
7.1 Performance
LCP &lt; 2.5s sur 4G mobile - images UploadThing via CDN, pas de resize Next.js
INP &lt; 200ms pour actions critiques (upload, création chantier)
next-intl : charger uniquement le namespace nécessaire par page (lazy loading des fichiers JSON)
React Query : staleTime 5 min pour liste chantiers, 30s pour photos

7.2 Scalabilité
Architecture serverless Vercel + Neon : scale automatique
Neon branching : branche DB isolée par PR de migration
i18n : fichiers JSON statiques servis depuis le CDN Vercel - zéro overhead DB
Microservice Factur-X : conteneur Docker sur Fly.io (auto-scale à 0)

7.3 Conformité RGPD multi-pays
Données artisan : EU-West (Neon Frankfurt, Vercel Paris)
CGU et politique de confidentialité localisées par pays (FR, DE, BE, ES)
Consentement cookies : bannière localisée - Cookieyes ou Axeptio
Droit à l'effacement : suppression cascade Prisma (onDelete: Cascade)
Registre de traitement : documenter le flux de données par pays
DPA (Data Processing Agreement) avec Vercel, Neon, UploadThing, Resend

7.4 Accessibilité
WCAG 2.1 niveau AA - shadcn/ui (Radix UI) gère ARIA, focus, keyboard nav
Contraste minimum 4.5:1 pour le texte courant
Attribut lang sur &lt;html&gt; mis à jour dynamiquement selon la locale active (géré par next-intl)
Textes alternatifs des images localisés

10  Contraintes &amp; notes pour LLM  - Section critique
10.1 Règles absolues - ne jamais violer
SÉCURITÉ DONNÉES : Chaque procédure Prisma qui lit des données artisan DOIT filtrer par artisanId: ctx.session.user.id. findMany() sans ce filtre est une faille critique. Ne jamais exposer stripeCustomerId, stripeSubscriptionId, User.locale, User.country, User.vatNumber ou tout champ financier dans une réponse tRPC publique.

PLAN CHECK : La vérification PRO se fait TOUJOURS côté serveur dans la procédure tRPC. Ne jamais lire le plan depuis une prop React ou un état client. Le plan peut changer via webhook Stripe en dehors de la session active.

STRIPE WEBHOOK : Ne JAMAIS mettre à jour le plan depuis l'URL de retour Stripe côté client. Toujours attendre checkout.session.completed côté serveur. Le client peut modifier les paramètres URL.

i18n - ZÉRO CHAÎNE EN DUR : Aucun texte d'interface dans le JSX/TSX. Toujours useTranslations() (Client Component) ou getTranslations() (Server Component). Les messages d'erreur TRPCError sont en anglais côté serveur - la traduction est la responsabilité du client.

MIDDLEWARE ORDRE : Dans src/middleware.ts, next-intl middleware s'exécute AVANT NextAuth. Le matcher doit exclure /api/*, /c/*, /admin, /_next, /.*\..*. La route /c/[token] ne doit jamais être derrière le middleware locale.

NEXTAUTH v5 : Utiliser auth() depuis @/server/auth.ts. Dans Server Components : await auth(). Dans tRPC context : ctx.session. Dans middleware : export { auth as middleware }. NE PAS utiliser getServerSession() (v4).

tRPC SERVER COMPONENTS : Les Server Components utilisent createCaller(ctx), PAS les hooks useQuery. Les hooks React Query sont client-only. Ne jamais importer api.project.list.useQuery() dans un fichier sans 'use client'.

MAGIC LINK SÉCURITÉ : getByToken() : select Prisma explicite. Retourner uniquement : id, name, status, artisanCompanyName, artisanLogoUrl, photos (url, note, takenAt), clientActions. JAMAIS artisanId, clientEmail, User.*, données financières.

MONTANTS EN DB : Tous les montants sont stockés en centimes (integer) dans la DB. Utiliser le helper formatCurrency(cents, currency, locale) pour l'affichage. Ne jamais stocker ni manipuler des floats pour de l'argent.

FACTUR-X / ZUGFERD : La génération se fait dans le microservice Python (lib factur-x), PAS dans Next.js. Le format dépend de User.country + User.vatScheme. ZUGFeRD DE = même microservice, paramètre country='DE'. Ne pas tenter de générer le XML EN 16931 en TypeScript.

10.2 Conventions de code
Nommage : camelCase variables/fonctions, PascalCase composants/types, SCREAMING_SNAKE constantes env
Fichiers : kebab-case sauf composants React (PascalCase.tsx)
Imports : alias @/ - jamais de chemins relatifs ../..
Traductions : clés de traduction en dot.notation : 'projects.status.ACTIVE', 'common.buttons.save'
Erreurs tRPC : TRPCError avec code approprié (NOT_FOUND, UNAUTHORIZED, FORBIDDEN, BAD_REQUEST)
Mutations : invalider le cache React Query après succès (utils.project.list.invalidate())
Prisma : transactions pour opérations multi-tables
Dates : toujours UTC en DB, Intl.DateTimeFormat pour l'affichage via helper formatDate()
Montants : toujours centimes en DB, formatCurrency() pour l'affichage
