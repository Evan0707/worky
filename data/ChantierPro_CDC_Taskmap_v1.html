01 Contexte & Vision produit

1.1 Problème identifié

Les artisans du bâtiment (plombiers, électriciens, carreleurs, peintres, maçons) gèrent leurs chantiers via des outils inadaptés : carnets papier, photos WhatsApp, SMS, tableurs Excel. Ce mode de fonctionnement génère :

  • Perte de temps administrative estimée à 6-8h par semaine
  • Litiges fréquents avec les clients (pas de trace écrite des étapes)
  • Aucun suivi de rentabilité par chantier (heures réelles vs estimées)
  • Non-conformité imminente à la facturation électronique obligatoire (sept. 2027)

1.2 Solution proposée

ChantierPro est un SaaS web-first (mobile-ready) permettant à un artisan de :

  • Gérer ses chantiers avec photos géolocalisées et horodatées
  • Partager l'avancement en temps réel avec ses clients via un magic link (sans compte client)
  • Saisir ses heures et matériaux pour générer automatiquement devis et factures
  • Être conforme à la facturation électronique Factur-X dès le lancement (avantage concurrentiel fort)

02 Utilisateurs & Rôles

2.1 Rôle Artisan (utilisateur principal)

L'artisan est le titulaire du compte. Il accède à l'application via le navigateur web (desktop ou mobile) après authentification. Il dispose de :

  • Création et gestion de ses chantiers
  • Upload de photos depuis son mobile (caméra ou galerie)
  • Saisie des heures travaillées et des matériaux par chantier
  • Génération de devis et factures PDF (premium)
  • Partage du chantier avec son client via magic link
  • Gestion de son abonnement et profil

2.2 Rôle Client (accès limité, sans compte)

Le client de l'artisan accède à une vue publique de son chantier via un magic link sécurisé. Il ne crée pas de compte. Il peut :

  • Voir les photos du chantier triées par date
  • Consulter l'avancement global (pourcentage, statut)
  • Lire les notes laissées par l'artisan
  • (Premium) Valider une étape et laisser un commentaire
  • (Premium) Signer électroniquement un devis

Note LLM — contrainte clé : Le client ne doit jamais pouvoir accéder aux données d'autres chantiers, même en manipulant l'URL. La sécurité repose sur l'unicité du share_token (UUID v4, 122 bits d'entropie) et les politiques Prisma qui filtrent toujours par share_token. Il n'y a pas de session client côté serveur.

2.3 Rôle Admin (backoffice interne)

L'administrateur (le développeur / fondateur) accède à un backoffice minimal protégé par une variable d'environnement ADMIN_SECRET pour :

  • Visualiser les métriques d'usage (inscriptions, conversions, churn)
  • Gérer manuellement les abonnements Stripe en cas d'erreur
  • Accéder aux logs d'erreurs Sentry

03 Fonctionnalités détaillées

3.1 Authentification (NextAuth v5)

Providers supportés

  • Email OTP (lien magique par email via Resend) — provider principal
  • Google OAuth 2.0 — provider secondaire

Comportements attendus

  • Session JWT stockée en cookie HttpOnly sécurisé (14 jours)
  • Middleware Next.js protège toutes les routes /(dashboard)/* sauf /c/[token]
  • Déconnexion côté serveur invalide le token JWT
  • Un utilisateur peut lier plusieurs providers au même compte

Contrainte LLM : Utiliser NextAuth v5 (Auth.js beta) et non v4. La configuration diffère significativement — notamment l'export auth() depuis auth.ts utilisé dans le middleware Edge. Ne pas utiliser getServerSession() de la v4.

3.2 Gestion des chantiers

Création d'un chantier

  • Champs requis : nom du chantier, adresse complète, nom du client
  • Champs optionnels : téléphone client, email client, date de début estimée, date de fin estimée, description libre
  • Génération automatique du share_token (UUID v4) à la création
  • Statut initial : ACTIVE

États d'un chantier

Limites freemium

  • Plan FREE : 3 chantiers ACTIVE ou PAUSED simultanément
  • Plan PRO : chantiers illimités
  • Tentative de création au-delà de la limite → modal d'upgrade Stripe

3.3 Gestion des photos

  • Upload depuis le navigateur (input file, drag & drop) ou caméra mobile
  • Compression automatique côté client avant upload (max 1200px côté long, qualité JPEG 80%) via browser-image-compression
  • Stockage sur UploadThing (S3-compatible) — clé unique par fichier
  • Métadonnées stockées en DB : url, key, note (texte libre), takenAt
  • Géolocalisation : récupérée via l'API Geolocation du navigateur si autorisée (lat, lng stockés)
  • Affichage en grille masonry triée par date décroissante
  • Pas de limite de photos en plan FREE (limite de 3 chantiers actifs suffit)

Note LLM — UploadThing : Utiliser le router UploadThing défini dans /api/uploadthing/core.ts. L'authentification est gérée par le middleware UploadThing qui lit la session NextAuth. Ne jamais uploader directement depuis le client sans passer par ce router.

3.4 Pointage heures & matériaux

Heures travaillées

  • Saisie : date, nombre d'heures (décimal, ex: 7.5), description optionnelle
  • Affichage : total d'heures par chantier, total par semaine
  • Taux horaire configurable dans les settings artisan (utilisé pour le calcul du devis)

Matériaux

  • Saisie : libellé, quantité, prix unitaire HT
  • Calcul automatique : sous-total par ligne, total matériaux HT
  • TVA configurée dans les settings (taux standard 20%, taux réduit 10%, franchise)

3.5 Magic link & espace client

Génération & partage

  • URL : https://app.com/c/[share_token]
  • Bouton "Partager avec le client" → bottom sheet avec options : copier le lien, SMS (sms: URI), email (mailto: URI), WhatsApp (wa.me)
  • Régénération du token : invalide immédiatement l'ancien lien
  • Expiration optionnelle : l'artisan peut définir une date d'expiration (null = jamais)

Vue client /c/[token] — plan FREE

  • Résolution du token : requête publicProcedure tRPC getByToken()
  • Si token invalide ou expiré : page d'erreur claire avec message explicite
  • Affichage : nom du chantier, statut, photos triées par date, notes artisan
  • Pas de session, pas de cookie, zéro donnée personnelle artisan exposée
  • PWA-ready : meta viewport, manifest.json, icône pour "Ajouter à l'écran d'accueil"

Vue client — plan PRO (fonctions supplémentaires)

  • Validation d'étape : bouton "Valider cette étape" → déclenche une notification push à l'artisan (Pusher)
  • Commentaire : champ texte libre (max 500 chars), pseudonyme optionnel
  • Signature électronique du devis : canvas tactile, sauvegarde en base64 PNG

3.6 Devis & Facturation (plan PRO)

Génération du devis

  • Pré-rempli automatiquement avec les heures et matériaux saisis
  • Champs éditables : désignation, quantité, prix unitaire HT, TVA par ligne
  • Ajout de lignes manuelles possibles
  • Calcul automatique : HT, TVA, TTC
  • Statuts devis : DRAFT, SENT, ACCEPTED, REFUSED

Génération de la facture

  • Basée sur le devis accepté ou créée manuellement
  • Numérotation séquentielle obligatoire (ex: 2026-001, 2026-002...)
  • Mentions légales obligatoires auto-remplies depuis le profil artisan
  • Export PDF via Edge Function Supabase (puppeteer headless ou react-pdf)
  • Statuts facture : DRAFT, SENT, PAID, OVERDUE
  • Relances automatiques : email à J+7 et J+14 si non payée (Resend)

Facture électronique Factur-X (plan PRO+)

  • Format : PDF/A-3 avec XML EN 16931 embarqué (profil MINIMUM ou EN16931)
  • Génération via la lib Python factur-x appelée depuis une Edge Function
  • Transmission via API d'une PDP partenaire agréée (ex: Pennylane API ou Indy)
  • L'artisan ne devient pas lui-même PDP — il délègue la transmission
  • Applicable uniquement aux factures B2B (pas B2C)

Contrainte LLM — Facturation élec. : Ne pas tenter de générer le format Factur-X côté Next.js/Node.js. Utiliser une Edge Function Python dédiée ou un microservice Python séparé car les libs Factur-X matures (factur-x, pdfplumber) sont Python. L'API Next.js appelle ce service via fetch interne.

4.5 Sécurité

  • Middleware Next.js : toutes les routes (dashboard) vérifiées via auth() de NextAuth v5 avant rendu
  • tRPC protectedProcedure : vérifie ctx.session — lève UNAUTHORIZED si absent
  • Isolation des données : chaque requête Prisma filtre TOUJOURS par artisanId = ctx.session.user.id
  • Magic link : token UUID v4, rate-limit 20 req/min par IP sur /api/trpc (middleware Edge)
  • Webhooks Stripe : vérification de signature via stripe.webhooks.constructEvent()
  • Upload : UploadThing vérifie la session NextAuth côté serveur, liste blanche des types MIME (image/jpeg, image/png, image/webp)
  • Variables d'environnement : jamais exposées côté client (pas de NEXT_PUBLIC_ pour les secrets)
  • CSP headers : configurés dans next.config.js

05 UX & Design system

5.1 Principes UX

  • Mobile-first : l'artisan utilise son téléphone sur chantier — tous les écrans doivent fonctionner à une main
  • 2 taps maximum : créer une photo ou un pointage d'heures ne doit jamais dépasser 2 actions depuis l'accueil
  • Offline-friendly : afficher les données en cache si le réseau est coupé (React Query staleTime élevé)
  • Progressive disclosure : ne montrer la complexité (facture, Factur-X) qu'au moment pertinent

5.2 Design system

  • Framework CSS : Tailwind CSS v4
  • Composants : shadcn/ui (Button, Card, Dialog, Form, Input, Table, Badge, Avatar, Sheet)
  • Typographie : Inter (Google Fonts) — pas d'autres polices
  • Palette principale : bleu #1A4F8A (primaire), vert #0F6E56 (succès), ambre #854F0B (warning), rouge #A32D2D (erreur)
  • Dark mode : supporté via next-themes dès le MVP
  • Icônes : Lucide React uniquement

06 Exigences non-fonctionnelles

6.1 Performance

  • LCP (Largest Contentful Paint) < 2.5s sur 4G mobile
  • INP (Interaction to Next Paint) < 200ms pour les actions critiques (upload, création chantier)
  • Les images sont servies via le CDN UploadThing — pas de resize côté Next.js
  • React Query : staleTime 5 min pour la liste des chantiers, 30 sec pour les photos

6.2 Scalabilité

  • Architecture serverless (Vercel + Neon) : scale automatique sans configuration
  • Neon branching : chaque PR de migration crée une branche DB isolée
  • Prisma : connection pooling via PgBouncer (Neon intégré)
  • UploadThing : pas de limite de stockage en plan payant

6.3 Conformité RGPD

  • Données artisan : stockées en EU-West (Neon Frankfurt, Vercel Paris)
  • Données client magic link : aucune donnée personnelle client en DB (pseudonyme optionnel uniquement)
  • Droit à l'effacement : suppression de compte supprime en cascade tous les projets, photos, entrées (onDelete: Cascade dans Prisma)
  • Politique de confidentialité et CGU obligatoires avant inscription
  • Cookies : session HttpOnly uniquement — pas de tracking analytics tiers sans consentement

6.4 Accessibilité

  • Standard visé : WCAG 2.1 niveau AA
  • shadcn/ui est basé sur Radix UI qui gère nativement l'accessibilité (ARIA, focus management, keyboard nav)
  • Contraste minimum 4.5:1 pour le texte courant

09 Contraintes & notes pour LLM

9.1 Règles absolues — ne jamais violer

Sécurité données : Chaque procédure Prisma qui lit des données artisan DOIT filtrer par artisanId: ctx.session.user.id. Ne jamais faire findMany() sans ce filtre. Ne jamais exposer stripeCustomerId, stripeSubscriptionId, ou le hash de session dans une réponse tRPC.

Plan check : La vérification du plan PRO doit toujours se faire côté SERVEUR dans la procédure tRPC (ctx.session.user.plan). Ne jamais se fier à une prop passée depuis le client ou à un état React côté client pour décider si une feature premium est accessible.

Stripe webhooks : Ne JAMAIS mettre à jour le plan utilisateur suite à un succès de checkout côté client (ex: URL de retour Stripe). Toujours attendre le webhook checkout.session.completed côté serveur. Le client peut modifier les paramètres URL de retour.

NextAuth v5 : Utiliser auth() depuis src/server/auth.ts (pas getServerSession). Dans les Server Components : const session = await auth(). Dans les procédures tRPC : ctx.session (injecté dans le context). Dans le middleware : export { auth as middleware } from '@/server/auth'.

tRPC Server Components : Les Server Components n'utilisent pas les hooks React Query. Utiliser le server caller : const caller = createCaller(context) puis await caller.project.list(). Ne pas importer les hooks api.project.list.useQuery() dans un Server Component — c'est un client-only pattern.

Magic link sécurité : La procédure publicProcedure getByToken ne doit retourner QUE les champs nécessaires à la vue client : id, name, status, photos (url, note, takenAt), timeEntries (agrégé), clientActions. Ne jamais inclure artisanId, clientEmail, ou données financières dans la réponse publique. Utiliser select Prisma explicitement.

9.2 Conventions de code

  • Nommage : camelCase pour variables/fonctions, PascalCase pour composants et types, SCREAMING_SNAKE pour les constantes env
  • Fichiers : kebab-case pour les fichiers, sauf les composants React (PascalCase.tsx)
  • Imports : alias @/ configuré dans tsconfig.json — ne pas utiliser de chemins relatifs ../..
  • Erreurs tRPC : toujours utiliser TRPCError avec le bon code (NOT_FOUND, UNAUTHORIZED, FORBIDDEN, BAD_REQUEST)
  • Mutations : toujours invalider le cache React Query après une mutation réussie (utils.project.list.invalidate())
  • Prisma : toujours utiliser des transactions pour les opérations qui modifient plusieurs tables
  • Env vars : valider via T3 Env (src/env.js) — le build échoue si une variable manque

9.3 Ce que ce document ne couvre pas (hors scope MVP)

  • Application mobile React Native (Expo) — phase post-validation web
  • Multi-artisans / équipes sur un même compte (feature Phase 2+)
  • Import/export comptable (FEC, connexion bancaire) — feature post-MRR stable
  • Marketplace de matériaux ou catalogues fournisseurs
  • Intelligence artificielle / reconnaissance de documents (devis photos)
  • API publique REST pour intégrations tierces
