I. ÉTAT DES LIEUX — CE QUI EXISTE
================================================================================

Modèle actuel :
  - 1 User peut créer 1 Team (OWNER) OU être membre d'1 Team (TeamMember)
  - Rôles : OWNER > ADMIN > MEMBER
  - Invitation par email (token + expiration 7j)
  - Les membres partagent les projets du OWNER via getArtisanContext()
  - Le plan PRO du owner est hérité par tous les membres

Schéma Prisma :
  - Team        : id, name, ownerId (unique), timestamps
  - TeamMember  : id, teamId, userId (unique), role, joinedAt
  - TeamInvitation : teamId, email, role, token, expiresAt, acceptedAt

Points forts :
  ✓ Data isolation correcte (artisanId = ownerId partout)
  ✓ PRO check server-side, hérité du owner
  ✓ Email d'invitation propre (React Email, 4 locales)
  ✓ Transaction atomique pour acceptation d'invitation
  ✓ UI claire avec badges de rôles


II. PROBLÈMES IDENTIFIÉS
================================================================================

A. ARCHITECTURE
───────────────

1. PAS DE PERMISSIONS GRANULAIRES
   - MEMBER voit tout, peut tout faire (créer/supprimer projets, factures, etc.)
   - ADMIN = MEMBER + peut inviter. Quasi aucune différence en pratique.
   - Aucun router ne vérifie le rôle du membre. getArtisanContext() renvoie
     juste artisanId — le rôle n'est jamais consulté dans les mutations.

   Exemple : un MEMBER peut supprimer un projet, créer une facture,
   supprimer des photos — exactement comme le OWNER.

2. PAS D'ATTRIBUTION DE PROJETS
   - Un membre voit TOUS les projets du owner.
   - Impossible d'assigner un membre à un chantier spécifique.
   - Sur un vrai chantier, un plombier ne doit pas voir les projets du maçon.

3. PAS DE TRAÇABILITÉ
   - Aucun champ "createdBy" ou "modifiedBy" sur les entités.
   - Impossible de savoir qui a ajouté une photo, pointé des heures, etc.
   - Critique pour la facturation : qui a travaillé combien d'heures ?

4. LIMITATION 1 TEAM PAR USER
   - Un artisan ne peut pas être membre de 2 équipes différentes.
   - Cas réel : un électricien intérimaire travaille pour 2 entreprises.
   - Bloquant pour la croissance.

5. PAS DE LIMITE DE MEMBRES
   - Le plan PRO ne limite pas le nombre de membres.
   - Un owner PRO pourrait inviter 100 personnes pour 15€/mois.
   - Pas de modèle économique viable pour les équipes.

B. UX
─────

6. DATE HARDCODÉE "fr-FR" DANS team-view.tsx
   - MembersCard et PendingInvitations utilisent
     new Intl.DateTimeFormat("fr-FR", ...) au lieu de formatDate(date, locale).

7. PAS DE VUE D'ACTIVITÉ DES MEMBRES
   - Le owner ne voit pas : qui a travaillé aujourd'hui, sur quel chantier,
     combien d'heures, quelles photos ont été prises.
   - Impossible de manager une équipe sans tableau de bord d'activité.

8. PAS DE NOTIFICATION EN TEMPS RÉEL
   - Pusher est configuré mais pas câblé pour l'équipe.
   - Quand un membre ajoute une photo ou des heures, le owner ne le sait pas.

9. OWNER DATA NON INCLUSE DANS team.get POUR LE OWNER LUI-MÊME
   - Le composant owner fait un cast: (team as any).owner
   - La query team.get (owned branch) n'include pas owner dans le select.
   - Fragile, peut casser silencieusement.

C. SÉCURITÉ
───────────

10. AUCUN CONTRÔLE D'ACCÈS PAR RÔLE DANS LES ROUTERS
    - project.delete, invoice.create, material.delete : aucun ne vérifie
      si le user est OWNER, ADMIN ou MEMBER.
    - Un MEMBER peut faire toutes les opérations destructrices.
    - C'est le problème le plus critique de tout le système.


III. REDESIGN PROPOSÉ
================================================================================

A. NOUVEAU MODÈLE DE PERMISSIONS
─────────────────────────────────

Principe : le rôle détermine ce que le membre peut faire.

  OWNER  : Tout. Gestion d'équipe, facturation, suppression.
  ADMIN  : Créer/modifier projets et factures. Inviter des membres.
           Ne peut pas supprimer l'équipe ni gérer l'abonnement.
  MEMBER : Voir les projets assignés. Ajouter photos, heures, matériaux.
           Ne peut pas créer/supprimer des projets ni des factures.

Implémentation technique :

  1. Modifier getArtisanContext() pour retourner aussi le rôle :
     → { artisanId, isTeamMember, teamId, role: TeamRole | null }

  2. Créer un helper requireRole() :

     function requireRole(
       ctx: Context,
       allowed: TeamRole[]
     ) {
       const { role } = ctx.teamContext;
       if (role && !allowed.includes(role)) {
         throw new TRPCError({ code: "FORBIDDEN" });
       }
     }

  3. Appliquer dans chaque router :

     Opération                  │ Rôles autorisés
     ───────────────────────────┼──────────────────
     project.create/delete      │ OWNER, ADMIN
     project.update             │ OWNER, ADMIN, MEMBER (si assigné)
     photo.upload               │ OWNER, ADMIN, MEMBER (si assigné)
     timeEntry.create           │ OWNER, ADMIN, MEMBER (si assigné)
     material.create            │ OWNER, ADMIN, MEMBER (si assigné)
     invoice.create/delete      │ OWNER, ADMIN
     invoice.sendReminder       │ OWNER, ADMIN
     invoice.markPaid           │ OWNER
     team.*                     │ (déjà géré)
     settings (billing/stripe)  │ OWNER


B. ASSIGNATION AUX PROJETS
───────────────────────────

Nouveau modèle Prisma :

  model ProjectAssignment {
    id        String   @id @default(cuid())
    projectId String
    userId    String
    assignedAt DateTime @default(now())
    project   Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)
    user      User     @relation(fields: [userId], references: [id])

    @@unique([projectId, userId])
    @@index([userId])
    @@index([projectId])
  }

Logique :
  - OWNER et ADMIN voient tous les projets.
  - MEMBER ne voit que les projets où il a un ProjectAssignment.
  - Modifier le query project.list :

    if (role === "MEMBER") {
      where.assignments = { some: { userId } };
    }

  - UI : sur la page projet, onglet "Équipe" avec bouton "Assigner un membre".


C. TRAÇABILITÉ (createdBy)
──────────────────────────

Ajouter un champ createdById optionnel sur :
  - Photo        : savoir qui a pris la photo
  - TimeEntry    : savoir qui a pointé
  - Material     : savoir qui a enregistré l'achat
  - Invoice      : savoir qui a créé la facture

  model Photo {
    ...
    createdById String?
    createdBy   User?   @relation("PhotoCreator", fields: [createdById], references: [id])
  }

Puis dans l'UI : afficher "Ajouté par Jean" à côté de chaque entrée.
Pour les heures : le owner voit un récap par membre dans le dashboard.


D. TABLEAU DE BORD D'ACTIVITÉ ÉQUIPE
─────────────────────────────────────

Nouvelle query : team.activity

  Retourne les 50 dernières actions de l'équipe :
  - Photos ajoutées (par qui, sur quel chantier)
  - Heures pointées (par qui, combien)
  - Matériaux ajoutés
  - Factures créées

  Implémentation : query sur Photo + TimeEntry + Material + Invoice
  triés par createdAt DESC, avec createdBy inclus.

  UI : Card "Activité récente" sur la page équipe ou le dashboard.


E. NOTIFICATIONS TEMPS RÉEL (PUSHER)
─────────────────────────────────────

Canaux Pusher par équipe :

  Channel : private-team-{teamId}

  Événements :
  - photo:added       → { projectName, memberName, count }
  - time:logged       → { projectName, memberName, hours }
  - material:added    → { projectName, memberName, label }
  - project:created   → { projectName, memberName }
  - member:joined     → { memberName }

  Côté serveur : après chaque mutation concernée,
  si isTeamMember → pusher.trigger(`private-team-${teamId}`, event, data)

  Côté client : toast discret "Jean a ajouté 3 photos sur Chantier Dupont"


F. PRICING ÉQUIPE
─────────────────

Le modèle actuel (PRO à 15€/mois pour membres illimités) n'est pas viable.

Proposition :

  Plan PRO Solo   : 15€/mois — 0 membre d'équipe (comme aujourd'hui)
  Plan PRO Équipe : 29€/mois — jusqu'à 5 membres
  Plan PRO+       : 49€/mois — membres illimités

  Ou plus simple : PRO inclut 2 membres, puis +5€/membre/mois.

  Implémentation :
  - Ajouter un champ maxTeamMembers sur User (ou calculé depuis le plan Stripe)
  - Vérifier dans team.invite :

    const currentCount = team.members.length;
    if (currentCount >= maxTeamMembers) {
      throw new TRPCError({ code: "FORBIDDEN", message: "Team member limit reached" });
    }
